# 前端设计文档

## Context

本文档描述"预案与实盘对比系统"前端的设计方案，基于后端 OpenSpec artifacts（proposal.md、design.md、各 spec.md）以及设计讨论中确认的前端决策。

### 后端上下文

- 后端：Java 17 + Spring Boot 3.4.3，REST API，JSON 响应包装在 `ApiResponse<T>` 中
- 数据库：MySQL + JPA，后端已完成全部 CRUD 和视图 API
- 主要 API：

| 端点 | 用途 |
|------|------|
| `GET/POST /api/plans` | 预案列表/创建 |
| `GET/PUT/DELETE /api/plans/{id}` | 预案详情/更新/删除 |
| `GET/POST /api/plans/{id}/conditions` | 条件列表/创建 |
| `GET /api/plans/{id}/executions` | 预案执行记录 |
| `GET/POST /api/actual-trades` | 实盘列表/创建 |
| `GET/PUT/DELETE /api/actual-trades/{id}` | 实盘详情/更新/删除 |
| `GET /api/views/period-summary?period=WEEK\|MONTH\|YEAR` | 视图 B：周期汇总 |
| `GET /api/views/holdings?refresh=` | 视图 C：当前持仓 |
| `GET /api/snapshots` | 历史快照查询 |
| `GET/PUT /api/system-config` | 系统配置（基准资金） |
| `GET/PUT /api/actual-account` | 实盘账户余额 |

响应格式统一为：

```json
{
  "success": true,
  "message": "...",
  "data": { ... },
  "timestamp": "2026-05-21T13:00:00"
}
```

### 已确认的前端设计决策

| 决策项 | 选择 |
|--------|------|
| 技术栈 | React + Vite + Tailwind CSS |
| 路由 | React Router |
| 数据获取 | React Query（TanStack Query） |
| 图表库 | Apache ECharts |
| 主题 | 默认深色 |
| 股票输入 | 纯文本框，无自动补全 |
| MA 触碰阈值 | 固定 0.3%，前端不暴露调节 |
| 对比维度 | 全局组合聚合（所有预案 vs 所有实盘） |
| 走势图 | 预案累计收益率 vs 实盘累计收益率，无基准线 |
| 数据补齐 | 非交易日（周末/节假日）前向填充（forward fill） |
| 默认首页 | 对比分析 — 走势图 |
| 目标平台 | 桌面端，不考虑响应式移动端 |
| 基准资金入口 | 首页常驻输入框，始终可见，始终可修改，无 onboarding 引导 |
| 实盘现金余额 | 仅在设置页展示，首页不展示 |
| 浮盈计算口径 | 预案侧 + 实盘侧独立计算，合并展示 |

## 组件清单

### 全局组件

#### `<AppLayout>`

页面框架：顶部栏 + 左侧导航 + 主内容区。

```tsx
<AppLayout>
  <Sidebar />
  <MainContent>{children}</MainContent>
</AppLayout>
```

#### `<Sidebar>`

左侧导航，含 5 个菜单项：对比分析、预案管理、实盘记录、历史快照、系统设置。使用 React Router `<NavLink>` 实现 active 状态。

#### `<TopBar>`

顶部栏，左侧标题，右侧操作按钮。刷新按钮触发 React Query 的 `invalidateQueries`。

#### `<StatusBadge>`

预案状态标签，4 种样式：

| 状态 | 标签文字 | 颜色 |
|------|---------|------|
| PENDING | 待触发 | 绿色 `#22C55E` |
| HOLDING | 持股中 | 橙色 `#F59E0B` |
| CLOSED | 已完成 | 蓝色 `#3B82F6` |
| EXPIRED | 已过期 | 灰色 `#6B7280` |

#### `<LoadingSkeleton>`

数据加载时的骨架屏，使用 Tailwind 的 `animate-pulse`。

#### `<EmptyState>`

无数据时的空状态展示，包含图标、文字和可选操作按钮。

#### `<ErrorAlert>`

接口错误时的提示，使用红色边框背景。

#### 基准资金输入框

- 位于 KPI 卡片上方，始终可见，始终可编辑
- 输入框默认显示当前 `baselineCapital` 值（从 `GET /api/system-config` 获取）
- 用户修改后点击"保存"触发 `PUT /api/system-config/baseline-capital`
- 保存后实时刷新 KPI 卡片和走势图数据
- **提高基准**时：显示提示"提高到 ¥X → 预案+实盘现金同步充值"
- **降低基准**时：显示警告"⚠️ 不可行，当前现金余额不足"，保存按钮禁用

#### KPI 卡片计算公式

```
预案组合收益率（%）= (预案现金 + 预案持仓市值 - baselineCapital) / baselineCapital × 100%
实盘组合收益率（%）= (实盘现金 + 实盘持仓市值 - baselineCapital) / baselineCapital × 100%
知行差（%）       = 预案组合收益率 - 实盘组合收益率

其中：
  预案持仓市值 = Σ(预案持股数 × 当前市价)
  实盘持仓市值 = Σ(实盘持股数 × 当前市价)
  数据来源：GET /api/views/holdings?refresh=true
```

#### 走势图数据处理

1. 调用 `GET /api/snapshots`，按 `snapshot_date` 分组
2. 每个日期计算预案组合收益率 `planReturn = (snap.planCashBalance + snap.planMarketValue - baselineCapital) / baselineCapital` 和实盘组合收益率 `actualReturn = (snap.actualCashBalance + snap.actualMarketValue - baselineCapital) / baselineCapital`
3. 转换为累计序列：`planSeries[i] = sum(planReturn[0..i])`，`actualSeries[i] = sum(actualReturn[0..i])`
4. 前向填充非交易日：周末/节假日使用最近一个交易日的数据
5. ECharts 渲染

#### 走势图 ECharts 配置要点

> 基准：`baselineCapital` 统一作为收益计算分母，snapshot 数据中包含各日期的 `planCashBalance`、`planMarketValue`、`actualCashBalance`、`actualMarketValue`。

```javascript
{
  tooltip: {
    trigger: 'axis',
    formatter: (params) => {
      const date = params[0].data[0];
      const planVal = params[0].data[1]?.toFixed(2) ?? '-';
      const actualVal = params[1]?.data[1]?.toFixed(2) ?? '-';
      const gap = (planVal - actualVal).toFixed(2);
      const color = gap >= 0 ? '#22C55E' : '#EF4444';
      return `<b>${date}</b><br/>
        预案: <b>${planVal}%</b><br/>
        实盘: <b>${actualVal}%</b><br/>
        差值: <span style="color:${color}">${gap >= 0 ? '+' : ''}${gap}%</span>`;
    }
  },
  legend: {
    data: ['预案累计收益', '实盘累计收益'],
    bottom: 0
  },
  xAxis: { type: 'time' },
  yAxis: {
    type: 'value',
    axisLabel: { formatter: '{value}%' }
  },
  dataZoom: [
    { type: 'inside', start: 0, end: 100 },
    { type: 'slider', start: 0, end: 100, height: 20 }
  ],
  series: [
    {
      name: '预案累计收益',
      type: 'line',
      smooth: true,
      itemStyle: { color: '#5470C6' },
      data: planSeries  // [date, cumulativeReturnPct] 二元组数组
    },
    {
      name: '实盘累计收益',
      type: 'line',
      smooth: true,
      itemStyle: { color: '#FE5C3A' },
      data: actualSeries
    }
  ]
}
```

配色：
- 预案线：`#5470C6`（蓝）
- 实盘线：`#FE5C3A`（橙）
- 正收益：`#22C55E`（绿）
- 负收益：`#EF4444`（红）

#### 表单行为

- `股票代码` 输入后不自动补全，用户自行确保代码正确
- `股票名称` 为可选项，用户可手动填写
- `周期` 切换时自动调整 `validUntil` 默认值：
  - 日度：默认当天（当日有效）
  - 周度：默认本周末（周五）
  - 月度：默认当月末
- `触发条件` 买入/卖出各**最多一条**，方向唯一（不能同方向两条条件）
- `触发容差 0.3%` 以灰色说明文字展示，**不可编辑**
- 买入/卖出条件通过独立的子表单区域管理，使用展开/折叠交互

#### 编辑页差异

- 编辑页仅对 `PENDING` 状态开放，非 PENDING 状态跳转回详情页并提示
- 条件管理在编辑页独立为条件列表 + 新增/编辑/删除操作

#### 提交逻辑

1. 校验必填字段
2. POST `/api/plans`，包含 `conditions` 数组
3. 成功后跳转到 `/plans/{id}`


此预览为只读说明，不影响提交逻辑（盈亏由后端计算）。

### 8. 系统设置

路径：`/settings`

```
┌──────────────────────────────────────────────────────────────┐
│  系统设置                                                   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  基准资金                                                   │
│  ───────────────────────                                   │
│  基准资金（元）  [500,000]  [保存]                         │
│                                                              │
│  💡 调整影响说明（实时计算）：                               │
│     • 提高到 ¥600,000 → 预案+实盘现金同步充值 +100,000     │
│     • 降低到 ¥400,000 → ⚠️ 不可行，当前现金余额不足        │
│       （需现金 ≥ ¥400,000 才能降低）                       │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│  预案账户                                                   │
│  ───────────────────────                                   │
│  当前现金余额        ¥483,200                               │
│  当前持仓市值        ¥  16,800                               │
│  总资产              ¥500,000  ← baselineCapital             │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│  实盘账户                                                   │
│  ───────────────────────                                   │
│  当前现金余额        ¥500,000  ← 恒等于基准资金             │
│                     （由系统自动同步，不可单独修改）          │
│                                                              │
│  [保存修改]                                                 │
└──────────────────────────────────────────────────────────────┘
```

#### 交互行为

- **基准资金输入框**：显示当前 `baselineCapital` 值，用户可直接编辑
- **实时影响说明**：根据输入值实时计算并显示调整影响
  - **提高基准**：`提示` 级别，显示"提高到 ¥X → 预案+实盘现金同步充值 +Y"
  - **降低基准**：
    - 若 `新基准 > PlanAccount.cashBalance`：警告级别，显示"⚠️ 不可行，当前现金余额不足"，保存按钮禁用
    - 若 `新基准 <= PlanAccount.cashBalance`：`提示` 级别，显示"降低到 ¥X → 预案+实盘现金同步回拨 -Y"
- **预案账户卡片**：展示 `PlanAccount.cashBalance`、`Σ持仓市值`、总资产（= baselineCapital）
- **实盘账户卡片**：展示 `ActualAccount.cashBalance`（恒等于 baselineCapital），标注不可单独修改
- **API 调用**：保存时调用 `PUT /api/system-config/baseline-capital`

#### API 请求

```
GET  /api/system-config          → 获取 baselineCapital
GET  /api/actual-account        → 获取 ActualAccount.cashBalance
GET  /api/views/holdings?refresh=true  → 获取持仓市值（计算总资产）
PUT  /api/system-config/baseline-capital  → 调整基准资金
```

#### 提交后行为

1. 调用 `PUT /api/system-config/baseline-capital`
2. 成功后自动刷新 KPI 卡片和走势图
3. 显示成功提示

## API 请求映射

### React Query 命名约定

```typescript
// Plans
useQuery({ queryKey: ['plans'], ... })           // GET /api/plans
useQuery({ queryKey: ['plan', id], ... })       // GET /api/plans/{id}
useMutation(createPlan)                          // POST /api/plans
useMutation(updatePlan)                          // PUT /api/plans/{id}
useMutation(deletePlan)                          // DELETE /api/plans/{id}

// Conditions
useQuery({ queryKey: ['plan-conditions', planId], ... })
useMutation(createCondition)
useMutation(updateCondition)
useMutation(deleteCondition)

// Executions
useQuery({ queryKey: ['plan-executions', planId], ... })

// Actual Trades
useQuery({ queryKey: ['actual-trades'], ... })
useQuery({ queryKey: ['actual-trade', id], ... })
useMutation(createActualTrade)
useMutation(updateActualTrade)
useMutation(deleteActualTrade)

// Views
useQuery({ queryKey: ['period-summary', period], ... })  // GET /api/views/period-summary
useQuery({ queryKey: ['holdings', refresh], ... })        // GET /api/views/holdings

// Snapshots
useQuery({ queryKey: ['snapshots', filters], ... })

// System Config
useQuery({ queryKey: ['system-config'], ... })           // GET /api/system-config
useMutation(updateBaselineCapital)                        // PUT /api/system-config/baseline-capital

// Actual Account
useQuery({ queryKey: ['actual-account'], ... })          // GET /api/actual-account
```

### 统一错误处理

通过 React Query 的 `onError` 回调统一处理 4xx/5xx 错误：

- 409 Conflict：状态不允许的操作（如编辑 HOLDING 预案）
- 404 Not Found：资源不存在
- 400 Bad Request：参数校验失败
- 500：后端异常

统一展示 `<ErrorAlert>` 并记录日志。

## 技术约束与边界

- 不实现用户认证（与后端 Non-Goals 保持一致）
- 不实现移动端响应式
- 不实现 PWA / 离线模式
- 不实现数据导出（PDF/Excel）
- 不实现 WebSocket 推送（页面刷新依赖 React Query 的自动轮询或手动刷新）
- 股票名称由用户手动输入，后端不做校验（股票名称仅用于展示）
- MA 触碰阈值固定 0.3%，前端不暴露配置项

---

## Non-Goals

（与后端 Non-Goals 保持一致）

- 不实现周期性自动重置的预案
- 不实现市价单、限价单等订单类型
- 不实现真实的证券账户开户、交易执行、资产托管
- 不实现用户认证/权限系统
- 不实现移动端适配
