## Context

当前项目为空白的 Spring Boot 骨架，无数据库、无 API。目标是从零构建一套完整的"预案-实盘对比系统"。

### 背景

用户需要一种机制来量化自己的交易策略与实际操作之间的差距。用户预先录入"预案"（买卖条件），系统在收盘后自动检查条件是否满足并模拟执行；同时用户在真实账户中操作实盘，系统记录实盘买卖。两套数据并行对比，揭示"知"与"行"之间的偏差。

### 技术约束

- **语言/框架**: Java 17 + Spring Boot 3.4.3（Maven）
- **现有依赖**: spring-boot-starter-web、actuator
- **需要新增**: JPA + MySQL、Redis（可选缓存）、Tushare API 集成
- **运行环境**: Docker 容器化（已有 docker-compose.yml）

## Goals / Non-Goals

**Goals:**

- 预案管理：创建、编辑、删除预案（PENDING 状态）
- 触发条件：支持 MA 均线触碰条件判断（买入/卖出方向）
- 实施记录：每触发一次生成一条 PlanExecution 记录
- 实盘录入：用户手动录入每笔真实买卖，独立计算收益
- 每日快照：每个交易日 17:00 自动生成快照，追踪收益变化
- 视图 B：按时间段（本周/本月）汇总对比
- 视图 C：当前持仓概览（浮盈/浮亏）

**Non-Goals:**

- 不支持周期性自动重置的预案（每次预案都是一次性的）
- 不支持市价单、限价单等订单类型
- 不实现真实的证券账户开户、交易执行、资产托管
- 不实现用户认证/权限系统（单用户本地使用）

## Decisions

### 决策 1: 数据库选型 — MySQL + JPA

**选择**: MySQL 数据库，通过 Spring Data JPA 访问。

**理由**: 用户已有 MySQL 环境，JPA 提供类型安全的实体管理和自动建表能力。

**替代方案**:
- H2 内存数据库: 仅适合开发测试，生产环境不持久化
- MongoDB: 适合文档型数据，但与现有 Spring Boot 生态集成不如 JPA 成熟
- SQLite: 不适合 Docker 容器化部署

### 决策 2: 预案触发检查机制 — 每日定时任务

**选择**: 每个交易日 17:00（UTC+8）执行检查，通过 Spring `@Scheduled` + `@EnableScheduling` 实现。

**理由**: 17:00 时当日 Tushare 数据已可用。定时任务比 API 实时调用更稳定，也避免了用户手动触发的不及时。

**替代方案**:
- API 实时触发: 触发延迟高，且用户可能忘记触发
- WebSocket 推送: 过度工程化，不需要实时性

### 决策 3: 收益计算时间基准 — 以买入执行日收盘价为准

**选择**: 预案收益计算基准为 `PlanExecution.triggerPrice`（即买入执行日的收盘价）。

**理由**:
- 预案触发时记录的是当天收盘价，作为买入成本价合理
- 实盘录入时用户填入的是自己的实际买入价（可能与收盘价不同）
- 两者以各自的买入价为基准计算当前收益，保证可比性

**替代方案**:
- 统一以 Tushare 日 K 线开盘价/收盘价为基准: 不够准确，因为实盘未必能在理想价位成交
- 以 Tushare 复权收盘价为唯一基准: 实盘收益计算会失真

### 决策 4: 触发条件类型 — 固定价格 + MA 均线，触碰型语义

**选择**: 触发条件分为两类，**触碰型语义**：

| 类型 | 买入条件 | 卖出条件 |
|---|---|---|
| 固定价格 | `close >= targetPrice` | `close <= targetPrice` |
| MA 均线 | `|close - MA(N)| / close <= 0.3%` | `|close - MA(N)| / close <= 0.3%` |

- **MA 周期**：支持 N = 5 / 10 / 20 / 60 / 120 / 250
- **触碰阈值**：固定 0.3%，不暴露给用户调节
- **去重逻辑**：仅首次触发有效。预案进入 HOLDING 状态后，条件再满足不重复触发（结合状态机自然去重）
- **日内波动展示**：`high` / `low` / `close` 直接取自 Tushare 日K线字段

**理由**: 触碰型语义允许价格在均线附近一定容差范围内即触发，比站上跌破更灵敏，0.3% 的固定阈值简洁且够用。

**替代方案**:
- 站上跌破语义（`close >= MA(N)` 或 `close <= MA(N)`）：需要价格明确在均线上方/下方才触发，灵敏度较低
- threshold 参数可调节：会使 UI 和 API 复杂度增加，当前阶段固定值足够

### 决策 5: 预案状态机 — 4 状态

**选择**: `PENDING → HOLDING → CLOSED`，加 `EXPIRED` 状态。

```
PENDING  ──买入触发──▶  HOLDING  ──卖出触发──▶  CLOSED
   │                       │
   └──有效期内未触发──▶ EXPIRED
```

**理由**:
- PENDING: 等待触发（可编辑）
- HOLDING: 持股中（触发后锁定，不可再编辑）
- CLOSED: 预案完整执行（买入+卖出都触发了）
- EXPIRED: 有效期内从未触发（可删除或归档）

### 决策 6: 快照策略 — 每个交易日末全量快照，保留 3 个月

**选择**: 每日 17:00 对所有活跃 Plan 和 ActualTrade 生成一条 `DailySnapshot`。默认保留 3 个月。

**理由**:
- 全量快照适合事后分析，可追溯任意一天的持仓状态
- 3 个月足以覆盖短期交易分析需求，同时控制数据量
- 可通过定期归档/清理超过 3 个月的快照数据

### 决策 7: 周期（有效期）语义 — 一次性窗口

**选择**: 周期参数仅决定有效窗口，不支持自动重置。

| 周期 | 有效期窗口 | 检查频率 |
|------|-----------|---------|
| 日度 | 仅创建当日当天 | 只检查一天 |
| 周度 | 本自然周（周一至五） | 每天都检查，直到触发或周五结束 |
| 月度 | 本自然月 | 每天都检查，直到触发或月末 |

### 决策 8: 全局基准资金与现金联动 — 两个账户同步联动

**选择**: `baselineCapital`（DECIMAL，默认 50 万）为全局唯一基准，同时管理两个账户的 `cashBalance`，两者均在基准调整时同步联动。

**初始值**: 用户首次使用系统时，若未手动录入实盘初始现金，`ActualAccount.cashBalance` 默认等于 `baselineCapital`（50 万）。

**现金余额维护方式**: 两个账户的 `cashBalance` 均**实时同步维护**，非事后推导：
- **PlanAccount.cashBalance**：每笔 PlanExecution 买入时自动扣减，卖出时自动回补。
- **ActualAccount.cashBalance**：每笔 ActualTrade 买入时自动扣减，卖出时自动回补。
- **买入校验**：买入时校验 `交易金额 <= cashBalance`，不足则拒绝触发/录入。

**基准调整联动规则**:
- **基准提高时**（如 50万 → 80万）：`PlanAccount.cashBalance` 和 `ActualAccount.cashBalance` 均同步充值至新的基准金额。
- **基准降低时**（如 80万 → 50万）：系统检测 `newBaseline <= PlanAccount.cashBalance`，若检测不通过，拒绝调整并返回错误。
- 历史快照数据**不重算**，新基准仅对调整后的新快照生效。

**理由**: 统一起点才能得出有意义的知行差百分比对比；现金只能回拨不能变现持仓，符合资金管理的常识逻辑。

**替代方案**:
- 两个账户各自独立的基准资金: 收益百分比各有基准，知行差无统一参照物
- 基准降低时允许变现持仓抵扣: 持仓市值为浮盈浮亏，不能当现金用
- 仅显示绝对金额不显示收益率: 预案和实盘规模可能不同，收益率更能反映策略质量

### 决策 9: 知行差计算方式 — 基于收益率，精确到小数点后两位

**选择**: `知行差 = 预案账户收益率% - 实盘账户收益率%`，以百分比而非绝对金额作为核心对比指标。

**公式**:
```
账户收益率 = (账户当前总市值 - baselineCapital) / baselineCapital × 100%
知行差     = planReturnPct - actualReturnPct
```

**账户总市值计算**:
```
总市值 = cashBalance + Σ(持仓股数 × 当前市价)
```

其中：
- `cashBalance`（现金余额）: 预案账户为虚拟现金，实盘账户为真实现金。两者均实时同步维护，随交易自动扣减/回补。
- `持仓市价`: 通过 Tushare 实时接口获取最新收盘价。
- `持仓股数`: 预案侧取自 PlanExecution 的 `quantity` 字段，实盘侧取自 ActualTrade 汇总。

**前端展示**:
```
预案收益率: +8.32%   实盘收益率: +6.10%   知行差: +2.22%
绝对金额:              预案账户 ¥2,840,000   实盘账户 ¥2,690,000
```

**理由**:
- 预案账户和实盘账户的初始资金规模可能不同（预案轻仓 vs 实盘重仓），收益率归一化后才有可比性。
- 绝对金额作为补充信息展示，满足用户"跑赢了多少万"的好奇心。
- 精确到小数点后两位，视觉上足够精细又不至于过度精确。

### 决策 10: 回滚容错 — 幂等保护

**选择**: 定时任务（MarketCloseTask）和预案执行（PlanExecution）均依赖状态机天然去重，不额外增加全局锁。

- **定时任务幂等**: 通过 `executed` 字段标记。检查 `executed == true` 则跳过本次执行，结合状态转移（PENDING→HOLDING→CLOSED）保证每个预案只触发一次。
- **多预案并发处理**: 在同一个事务内批量处理，同一预案不会被重复调度。

**理由**:
- 状态机的状态转移天然约束了执行次数，不需要额外分布式锁
- PlanExecution 的 `executed` 字段 + 状态转移双向保证，执行过一次就不会再执行

## Data Model

### SystemConfig（系统配置）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| baselineCapital | DECIMAL(18,2) | 全局基准资金，单位元，默认 500000 |
| updatedAt | DATETIME | 最后修改时间 |

### PlanAccount（预案账户）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| cashBalance | DECIMAL(18,2) | 虚拟现金余额，初始等于 baselineCapital，随 PlanExecution 买卖实时扣减/回补 |
| updatedAt | DATETIME | 最后更新 |

### ActualAccount（实盘账户）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| cashBalance | DECIMAL(18,2) | 真实现金余额，初始值由用户手动录入（未录入时默认等于 baselineCapital），随 ActualTrade 买卖实时扣减/回补 |
| updatedAt | DATETIME | 最后更新 |

> 两个账户的 `cashBalance` 均实时同步维护。买入时校验 `交易金额 <= cashBalance`，不足则拒绝。基准调整时两者同步联动，历史快照不重算。

## Risks / Trade-offs

| Risk | Severity | Mitigation |
|------|----------|------------|
| Tushare 数据延迟或不可用 | 中 | 定时任务增加重试机制，记录失败日志 |
| 实盘滞后录入导致快照数据不连续 | 低 | 快照生成时，实盘侧按最近录入的持仓计算 |
| 预案卖出触发后实盘还在持股 | 低 | 预案侧和实盘侧完全独立计算，时间线不同步是正常预期 |
| 数据量大导致快照表膨胀 | 中 | 定期归档/清理超过 3 个月的快照数据 |
| 基准资金降低时现金不足 | 低 | 后端增加 `newBaseline <= cashBalance` 可行性检测，不通过则拒绝调整并返回明确错误信息 |

## Migration Plan

1. **Phase 1（数据库初始化）**: 添加 JPA 依赖，配置 MySQL 连接，JPA auto-ddl 自动建表
2. **Phase 2（核心实体 + CRUD）**: 实现 Plan、PlanCondition、ActualTrade 实体及 REST API
3. **Phase 3（定时任务）**: 实现每日 17:00 的触发检查 + 快照生成
4. **Phase 4（视图 API）**: 实现视图 B/C 的聚合查询接口
5. **Phase 5（集成）**: 对接 Tushare 收盘价数据，完成端到端联调

回滚策略: 使用 Docker volume 挂载 MySQL 数据目录，通过 docker-compose down 保留数据。
